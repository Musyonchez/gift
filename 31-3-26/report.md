## 1. Introduction

Nairobi's matatu network is one of the most complex urban transport systems in East Africa, serving millions of commuters daily across hundreds of informal routes. Finding efficient routes within this network is a real-world problem well-suited to AI search algorithms. This project models a simplified version of the matatu network as a weighted graph and applies three search algorithms — **BFS**, **DFS**, and **A\*** — to find routes between stops, comparing their efficiency and suitability for Nairobi's dynamic traffic environment.

---

## 2. Graph Model

### 2.1 Network Design

The matatu network is modelled as an **undirected weighted graph**:
- **Nodes (15 stops):** CBD, Westlands, Parklands, Eastleigh, TRM, Kasarani, South B, South C, Langata, Karen, Rongai, Ngong Road, Embakasi, Donholm, Kikuyu, Ngong Town
- **Edges (24 connections):** each edge carries a weight representing approximate route distance in kilometres
- **Coordinates:** each node has real latitude/longitude for the A\* heuristic

### 2.2 Representation

```
graph = {
  "CBD":       [("Westlands", 5), ("Parklands", 3), ("Eastleigh", 4), ...],
  "Westlands": [("CBD", 5), ("Parklands", 2), ("Kikuyu", 16), ...],
  ...
}
```

An adjacency list was chosen because the graph is **sparse** (average ~3 connections per node), making it more memory-efficient than an adjacency matrix.

## 3. Algorithm Descriptions

### 3.1 Breadth-First Search (BFS)

BFS explores all neighbours at the current depth level before moving deeper, using a **FIFO queue**.

- **Completeness:** Yes — always finds a path if one exists
- **Optimality:** Optimal for unweighted graphs (fewest hops); does **not** guarantee minimum cost in weighted graphs
- **Time complexity:** O(b^d) where b = branching factor, d = depth
- **Space complexity:** O(b^d) — stores entire frontier

**How it works in this context:** BFS expands stops level by level outward from the source. It will find the route with the fewest matatu changes, even if it is not the shortest by distance.

---

### 3.2 Depth-First Search (DFS)

DFS explores as far as possible down one branch before backtracking, using a **LIFO stack**.

- **Completeness:** Yes (in finite graphs without cycles, with visited set)
- **Optimality:** **No** — returns the first path found, not the shortest
- **Time complexity:** O(b^m) where m = maximum depth
- **Space complexity:** O(bm) — only stores current path + stack, very memory efficient

**How it works in this context:** DFS dives deep into one route direction immediately. It may find a very long or roundabout route but uses minimal memory — useful when memory is constrained.

---

### 3.3 A\* Search (Informed)

A\* selects the next node using **f(n) = g(n) + h(n)**:
- **g(n):** actual cost from start to node n (km travelled so far)
- **h(n):** heuristic estimate from n to goal — **straight-line distance** in km

The straight-line distance heuristic is **admissible** (never overestimates true cost) because roads are always ≥ straight-line distance. This guarantees A\* finds the optimal path.

- **Completeness:** Yes
- **Optimality:** Yes — guaranteed with admissible heuristic
- **Time complexity:** O(b^d) in the worst case, but typically much better
- **Space complexity:** O(b^d) — stores all generated nodes

**Heuristic formula:**
```
Δlat_km = (lat_goal - lat_node) × 111
Δlon_km = (lon_goal - lon_node) × 111 × cos(mean_lat)
h(n) = sqrt(Δlat_km² + Δlon_km²)
```

## 5. Analysis — Suitability for Kenyan Traffic

### 5.1 Why A\* is Best for Nairobi

Nairobi traffic is highly variable — the same 5 km route can take 10 minutes or 2 hours depending on time of day. A\* is the most suitable algorithm because:

1. **It minimises total distance** (proxy for cost), which is the primary concern when planning a matatu journey
2. **The straight-line heuristic is effective** in Nairobi's roughly radial road layout from CBD outward
3. **It expands fewer nodes** than BFS, making it faster — important in real-time applications (e.g., a mobile routing app)

### 5.2 Limitations of BFS and DFS

**BFS** treats all edges as equal weight when selecting the next level to explore. In a matatu context, this means it might suggest a 3-hop route of 25 km over a 2-hop route of 15 km — clearly suboptimal.

**DFS** is unpredictable for routing — it may find a very roundabout path and gives no guarantee of quality. It is better suited to problems like maze solving where completeness matters more than optimality.

### 5.3 Traffic Variability Consideration

A static graph (fixed edge weights) is a simplification. In reality:
- Peak hour: CBD → Westlands can take 45 min (effectively "farther")
- Off-peak: same route takes 10 min

A more realistic model would use **time-dependent edge weights** or **dynamic re-weighting** based on real-time traffic data (e.g., from Google Maps API or Nairobi County traffic sensors). A\* is most easily extended to handle this, as only the edge costs need updating — the algorithm itself remains unchanged.

## 6. Ethical & Societal Reflection

### 6.1 Urban Mobility and Equity

Route optimisation algorithms can inadvertently reinforce inequality. If an app consistently recommends routes that only pass through well-served areas (Westlands, Karen), it may **underserve peripheral estates** like Mathare, Kayole, or Mukuru — where matatu coverage is already thin.

### 6.2 Data Bias

The graph in this project uses approximate distances from major stops. Real matatu networks include hundreds of informal stops that are **undocumented and unmapped** — a form of data invisibility. Any routing system built on incomplete data would give worse recommendations to commuters in informal settlements.

### 6.3 Digital Divide

A smartphone-based matatu routing app would benefit middle-class commuters more than the majority who use feature phones or have no data bundles. Nairobi's **USSD-based services** (no internet needed) would be more equitable — this is a real design consideration for any AI transport tool.

### 6.4 Environmental Impact

Routing passengers efficiently reduces the number of half-empty matatus making extra loops. This is a genuine environmental benefit — fewer emissions per passenger-km — which aligns with Nairobi's ongoing efforts to reduce urban air pollution.

---

## 7. Limitations

1. **Simplified network** — 15 nodes vs. hundreds of real Nairobi stops
2. **Static edge weights** — no real-time traffic data
3. **Undirected graph** — some matatu routes are one-way in practice
4. **No transfer cost** — switching matatus has a time and cost penalty not modelled here
5. **Heuristic accuracy** — straight-line distance assumes Euclidean space; Nairobi's hilly terrain means actual road distances deviate more than expected

---

## 8. Conclusion

This project demonstrates that **A\*** significantly outperforms BFS and DFS for Nairobi matatu routing: it consistently finds the optimal (lowest cost) path while expanding the fewest nodes. BFS is useful when minimising transfers matters more than distance. DFS, while memory-efficient, is unsuitable for optimal routing.

The work highlights that real-world AI deployment in Kenyan transport must account for data gaps, equity of access, and traffic variability — not just algorithmic correctness.
